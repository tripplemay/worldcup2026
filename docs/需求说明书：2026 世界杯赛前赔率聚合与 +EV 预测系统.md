# 需求说明书：2026 世界杯赛前赔率聚合与 +EV 预测系统

## 1. 项目概述 (Project Overview)

本项目旨在构建一个针对 2026 美加墨世界杯的体育赛前数据分析系统。系统通过定时抓取多平台的博彩赔率，结合基于历史射门数据的 xG（预期进球）泊松分布预测模型，自动计算“公平赔率”，并扫描全网寻找正期望值（+EV）的投注或套利机会。

## 2. 技术栈架构 (Technology Stack)

- **前端 (Frontend):** React + Tailwind CSS，采用 Horizon UI PRO 面板架构进行数据可视化。
    
- **后端 (Backend):** FastAPI (Python 3.10+) 提供高性能异步 RESTful API 服务。
    
- **数据库 (Database):** PostgreSQL 作为核心关系型存储（保存球队映射、历史战绩、赔率快照）。
    
- **数据源 (Data Sources):** * 赔率源：The Odds API (REST)
    
    - 战绩源：本地或第三方历史比赛统计（需包含射正、射偏数据）
        
- **部署 (Deployment):** Docker 容器化，适配 AWS ECS 环境运行。
    

## 3. 核心业务模块设计 (Core Modules)

### 3.1 数据摄取管道 (Data Ingestion Pipeline)

- **赔率抓取任务 (Cron Job):**
    
    - 目标 API：The Odds API `/v4/sports/soccer_fifa_world_cup/odds/`
        
    - 参数配置：`regions=eu,us,uk`, `markets=h2h,spreads,totals`, `oddsFormat=decimal`。
        
    - 执行频率：距离开赛 > 24 小时，每 4 小时一次；距离开赛 < 24 小时，每 1 小时一次。
        
- **实体解析 (Entity Resolution):**
    
    - 必须实现一个标准化映射字典（Team Name Mapping），解决数据源与 API 之间国家队命名的不一致（如 "USA" vs "United States"）。所有入库操作前必须通过该字典洗出标准 `team_id`。
        

### 3.2 预测算法引擎 (Prediction Engine)

系统不依赖外部预测，采用自建的 **Bottom-Up 统计学模型**，分为两步：

1. **xG（预期进球）计算器:**
    
    - 基于宏观射门数据转化为 xG 的加权公式。
        
    - **计算公式:** `xG = (Shot_on_Target * 0.30) + (Shot_off_Target * 0.05)`
        
    - 通过对球队近期（如近 10 场）比赛的 xG 进行指数加权移动平均（EWMA），得出该球队的“场均创造 xG”和“场均丢失 xG”。
        
2. **泊松分布与公平赔率推导 (Poisson Distribution):**
    
    - 计算主队本场预期进球（Lambda）和客队本场预期进球（Mu）。
        
    - 使用泊松分布公式构建进球概率矩阵（0-5球），累加得出：主胜概率（Home Win %）、平局概率（Draw %）、客胜概率（Away Win %）。
        
    - **公平赔率 (Fair Odds) = 1 / 对应赛果概率**。
        

### 3.3 价值扫描与报警 (+EV Scanner)

- 提取预测引擎输出的“公平赔率”（剥离了庄家抽水的理论公允价值）。
    
- 对比数据库中最新抓取的各大 Soft Books（如 Bet365, BetMGM 等）的赔率。
    
- **期望值计算公式:** `EV = (真实概率 * 博彩公司赔率) - 1`
    
- **触发条件:** 当某家平台的 `EV > 0.02`（即 2% 以上的正期望值）时，在前端 Dashboard 高亮标记该赛事、对应的平台及盘口。
    

## 4. 数据库实体设计 (Database Schema Outline)

|**表名 (Table)**|**字段说明 (Fields)**|**用途**|
|---|---|---|
|`teams`|`id`, `standard_name`, `api_name`, `stats_name`|球队实体映射表，解决重名对齐问题|
|`historical_matches`|`match_id`, `date`, `team_home_id`, `team_away_id`, `sot_home`, `sot_away`, `soff_home`, `soff_away`, `calculated_xg_home`, `calculated_xg_away`|存储历史战绩及洗出的单场 xG|
|`team_ratings`|`team_id`, `current_xg_for`, `current_xg_against`, `updated_at`|缓存球队最新动态实力池（EWMA 结果）|
|`odds_snapshots`|`id`, `match_id`, `bookmaker`, `market_type`, `odds_home`, `odds_draw`, `odds_away`, `timestamp`|赔率快照表，用于比对和追踪赔率走势|

## 5. API 接口定义 (API Endpoints Specification)

后端 (FastAPI) 需优先实现以下核心接口供前端调用：

- `GET /api/v1/matches/upcoming`
    
    - 功能：获取即将进行的赛事列表，包含基础对阵信息和开赛时间。
        
- `GET /api/v1/predictions/{match_id}`
    
    - 功能：返回特定赛事的 xG 预测结果、胜平负概率及模型推导的公平赔率。
        
- `GET /api/v1/scanner/ev-opportunities`
    
    - 功能：返回当前全网所有 EV > 0 的套利/价值投注机会列表。数据结构需包含：比赛、博彩公司名称、盘口方向、当前赔率、模型期望值(EV)。
        
- `POST /api/v1/engine/trigger-xg-calc`
    
    - 功能：内部触发接口，强制基于最新的 `historical_matches` 数据重新计算所有球队的 `team_ratings`。
        

## 6. 开发优先级安排 (Implementation Priorities)

- **Phase 1:** 搭建 FastAPI 骨架与 PostgreSQL 数据库。跑通 The Odds API 的数据摄取与定时任务，实现前端表格基础展示。
    
- **Phase 2:** 导入战绩数据，编写 Python xG 转换与泊松分布推理算法。实现预测接口，将“公平赔率”跑通。
    
- **Phase 3:** 整合前后端，实现 EV Scanner 业务逻辑，在 Horizon UI 仪表盘中完成 +EV 机会的高亮渲染与交互。
